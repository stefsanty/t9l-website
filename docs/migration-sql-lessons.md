# Migration SQL lessons — post-mortem v1.86.0

## The incident (2026-05-09)

### What broke

`prisma migrate deploy` ran during the v1.86.0 prod deploy. The migration `20260510000000_plm_preferred_secondary_positions` failed immediately with:

```
Database error code: 42P01
ERROR: relation "PlayerLeagueMembership" does not exist
```

### Root cause

The migration SQL was hand-authored and used the **Prisma model name** (`"PlayerLeagueMembership"`) instead of the **`@@map`-ed SQL table name** (`"PlayerLeagueAssignment"`):

```sql
-- WRONG (what shipped):
ALTER TABLE "PlayerLeagueMembership" ADD COLUMN ...
UPDATE "PlayerLeagueMembership" SET ...

-- CORRECT (what it should have been):
ALTER TABLE "PlayerLeagueAssignment" ADD COLUMN ...
UPDATE "PlayerLeagueAssignment" SET ...
```

The Prisma schema has `model PlayerLeagueMembership { ... @@map("PlayerLeagueAssignment") }`. Prisma's TypeScript client uses the model name (`PlayerLeagueMembership`); the actual Postgres table is `PlayerLeagueAssignment`. Prisma's auto-generated migrations use the `@@map` name. Hand-authored SQL does not.

This was even documented in `20260507100000_membership_alpha`:
> "the model rename `PlayerLeagueAssignment` → `PlayerLeagueMembership` is via Prisma `@@map` so the SQL table name stays unchanged (zero DDL rename)"

### The cascade

1. Migration failed at the `ALTER TABLE` step — no schema change, no data change.
2. Prisma wrote an error row to `_prisma_migrations` (`finished_at = null`, `applied_steps_count = 0`, error log populated).
3. Prisma's error state **blocks all future `migrate deploy` calls** — "new migrations cannot be applied before the error is recovered from."
4. The app code (v1.86.0) references `preferredPositions` / `secondaryPositions` on the `PlayerLeagueMembership` Prisma model. Those columns don't exist yet → every query touching PLM returned a DB error → "No leagues yet" / ERROR 941359833.
5. Vercel auto-rollback reverted the code deploy. Underlying schema and data were untouched throughout. After rollback, the app served correctly from the pre-v1.86.0 code against the unchanged schema.

### Recovery steps

1. `prisma migrate resolve --rolled-back 20260510000000_plm_preferred_secondary_positions` — cleared the error block from `_prisma_migrations`. **Only prod write performed.**
2. Corrected migration SQL to use `"PlayerLeagueAssignment"` in both statements.
3. Shipped corrected migration as v1.86.1.

## The lesson: NEVER hand-author migration SQL

Always use `prisma migrate dev --create-only` to generate migration SQL:

```bash
npx prisma migrate dev --create-only --name <descriptive_name>
```

This generates a migration file in `prisma/migrations/` with SQL that correctly references `@@map`-ed table names, correct enum casts, and correct constraint naming. The file is left un-applied locally so you can review it before running.

**Why hand-authoring fails:**
- `@@map` names diverge from model names silently — Prisma resolves this, your text editor does not.
- Enum casts (`"MyEnum"`) must match the Postgres type name, which Prisma also derives from `@@map` / `@map`.
- Constraint and index names follow Prisma's naming convention — hand-authoring those introduces drift that future auto-generated migrations will try to reconcile.

**Permitted exceptions:** comments, `-- rollback:` annotations, and pure-data `UPDATE`/`INSERT` statements that don't reference table or column names structurally. Even then, table names in those DML statements must use the `@@map` name.

## Checklist before shipping any migration

- [ ] Migration SQL was generated by `prisma migrate dev --create-only`, NOT hand-authored from scratch.
- [ ] If any table or column name was typed by hand, verify it against `@@map` / `@map` directives in `prisma/schema.prisma`.
- [ ] Migration does not contain `DROP TABLE`, `TRUNCATE`, or `DELETE FROM` outside the initial `new_schema` migration (which is idempotent by design).
- [ ] `prisma migrate status` returns "up to date" on a local DB after applying the migration.
